\begin{abstract}
Zero-suppressed binary decision diagrams (ZDDs) compactly represent families of
combinatorial solutions, and it is well known that building a ZDD along a path
decomposition of small width keeps it small. Solvers for combinatorial
reconfiguration, however, use a ZDD in a more demanding way: they first build a
ZDD of the entire solution space and then repeatedly rebuild ZDDs as the search
advances, one transition step at a time. Whether, and how much, a small-width
decomposition helps such a solver overall has been unclear, once the cost of
these repeated computations and of computing the decomposition itself is taken
into account.
We study path decomposition as a preprocessing step for \textsc{ddreconf}, a
state-of-the-art ZDD-based solver for independent set reconfiguration under the
token jumping rule. The preprocessing only reorders the input edges along a
small-width decomposition; the solver itself is left untouched. On both the
shortest and the farthest variant of the problem, the reordering consistently
reduces ZDD size, memory, and running time, and lets the solver clear more
instances within the time limit. Computing an optimal decomposition turns out to
be unnecessary: a fast heuristic reaches essentially the same width and the same
gains, and on large instances, whose optimal pathwidth is out of reach, it is the
only option that lets the solver finish at all. Tracing the effect of width
through the solver, we find that it shapes not only the solution-space ZDD but
also the intermediate ZDDs rebuilt at every step, so its benefit extends across
the whole search; and the width of the decomposition, rather than the size of the
graph, turns out to be a strong predictor of whether an instance can be solved.
\end{abstract}

\section{Introduction}
Combinatorial reconfiguration asks how one feasible solution of a combinatorial
problem can be turned into another through a sequence of small local changes, with
every intermediate step also feasible~\cite{nishimura18}. A prototypical example
is independent set reconfiguration (\ISR): given two independent sets of a graph,
can one be transformed into the other by moving the tokens placed on its vertices,
keeping an independent set throughout? We adopt the token jumping rule, in which
each step moves a single token to another vertex, equivalently, swaps one vertex
of the current independent set for a vertex outside it. \ISR is \PSPACE-complete in
general~\cite{kamemi12,itdehapasiueun11}, and building solvers that are efficient
in practice remains an active line of work.

Figure~\ref{fig:core-example} illustrates a benchmark instance from the CoRe
Challenge 2023~\cite{sotaokit24}: each node of the solution-space graph on the
right is an independent set of size $3$, and two nodes are adjacent when one can
be reached from the other by a single token jump.

To address this computational hardness, various practical approaches have been
developed, including Answer Set Programming (ASP)-based
solvers~\cite{ybisu24} and AI-planning techniques~\cite{cherkamupeposesisp23},
both of which excel at finding short reconfiguration sequences. Another line of
work uses zero-suppressed binary decision diagrams (ZDDs)~\cite{Minato1993}, compact data structures
for implicitly representing families of sets. By using ZDDs to compress
structural regularity in the solution space, ZDD-based solvers provide a
complementary approach and can sometimes outperform ASP and planning methods,
especially on instances whose solution spaces admit compact ZDD representations~\cite{ItoKNSSTT23}.
When a ZDD is built from a graph, its size is
closely tied to the pathwidth of the graph: roughly speaking, the smaller the
pathwidth, the smaller the ZDD~\cite{inoue16}. This connection is the basis of a
standard speedup in many ZDD applications, namely ordering variables along a
small-width path decomposition to keep the constructed ZDD small.


%Reconfiguration solvers, however, use a ZDD in a two-stage way that complicates
In such solvers, this representation is used in a two-stage way, which complicates
the picture. They first construct a ZDD representing the entire solution space,
and then repeatedly apply transition operations to the diagram, advancing the search one
step at a time until a reconfiguration sequence is found. A small-width
decomposition is known to help the first stage, the construction of the
solution-space ZDD. It has remained unclear how much it helps the solver as a
whole, once one also accounts for the cost of the repeated transition computations
and for the cost of computing the decomposition in the first place.



We study path decomposition as a preprocessing step for
\textsc{ddreconf}~\cite{ItoKNSSTT23}, an implementation of a state-of-the-art ZDD-based
solver for \ISR
under the token jumping rule. \textsc{ddreconf} reads a graph as an ordered list
of edges and processes them in that order; this order fixes the path
decomposition the solver implicitly follows, and hence the sizes of the ZDDs it
builds. Our preprocessing simply reorders the edges along a path decomposition of
small width and leaves the solver itself untouched. We evaluate the idea on two
variants that probe the solution space in complementary ways: the shortest
variant, which looks for a shortest reconfiguration sequence between two given
independent sets, and the farthest variant, which looks for an independent set as
far as possible from a given one. Our contributions are as follows.
\begin{itemize}
\item We show experimentally that reordering the input edges along a path
decomposition consistently improves \textsc{ddreconf}: it shrinks the
solution-space ZDD, lowers running time and memory, and increases the number of
instances solved within the time limit, on both the shortest and the farthest
variant.
\item We show that an optimal decomposition is unnecessary in practice. A fast
heuristic decomposition reaches essentially the same width as the optimal one and
brings the same gains; moreover, on large instances whose optimal pathwidth cannot
be computed in reasonable time, the heuristic is the only one of the two that lets
the solver reach them at all.
\item We trace where the gains come from and find that the decomposition width
shapes not only the solution-space ZDD but also the intermediate ZDDs that are
rebuilt at every transition step, so its effect extends across the whole search.
The exceptions form a recognizable class of structured instances whose pathwidth
is a loose upper bound on the actual ZDD size.
\item We find that the width of the decomposition the solver works with, rather
than the size of the input graph, is a strong predictor of whether an instance can
be solved.
\end{itemize}

Section~\ref{sec:conclusion} concludes.


\section{Preliminaries}
\label{sec:preliminaries}

\subsection{Independent Set Reconfiguration}
\label{subsec:isr}

Let $G=(V,E)$ be an unweighted and undirected graph, where $V$ is the set of vertices and $E$ is the set of edges. 
A vertex subset $I\subseteq V$ is an \emph{independent set of $G$} if no two vertices in $I$ are adjacent in $G$.
For a positive integer $k$, the \emph{solution space} $\mathcal{S}_k(G)$ is a graph whose nodes are the independent sets of $G$ with size $k$; two nodes are joined by an edge in $\mathcal{S}_k(G)$ if and only if the corresponding independent sets $I$ and $I'$ satisfy $|I \setminus I'| = |I' \setminus I| = 1$. (See \figurename~\ref{fig:core-example}.) 
Then, a path in $\mathcal{S}_k(G)$ connecting two nodes is called a \emph{reconfiguration sequence} between them, and the \emph{length} of the reconfiguration sequence is defined to be the number of edges in the path. 
The \emph{distance} between two nodes in $\mathcal{S}_k(G)$ is the length of a shortest reconfiguration sequence between them;
or it is $+\infty$ if no such sequence exists. 

Given a graph $G$ and two independent sets $I_{\ini}$ and $I_{\tar}$ of $G$ such that $|I_{\ini}|=|I_{\tar}| = k$, the \emph{Independent Set Reconfiguration} (\ISR) problem under the \emph{Token Jumping} (TJ) rule is to determine whether there exists a reconfiguration sequence between $I_{\ini}$ and $I_{\tar}$ in $\mathcal{S}_k(G)$.
Note that \ISR is a decision problem, and hence we are not asked for an actual reconfiguration sequence as an output. 
This problem is known to be \PSPACE-complete~\cite{kamemi12}, even when restricted to graphs of bounded pathwidth~\cite{Wrochna18}.
The \PSPACE-completeness implies that there are yes-instances such that even a
shortest reconfiguration sequence has super-polynomial length, unless $\PSPACE = \NP$.

In this paper, we study two variants of \ISR under \TJrule which probe the structure of the solution space in complementary ways.

\paragraph{Shortest variant} 
Given an \ISR instance $(G, I_\ini, I_\tar)$, the \emph{shortest variant} aims to
determine the existence of a reconfiguration sequence between $I_\ini$ and
$I_\tar$ and, if so, to compute the distance between them. 
This variant generalizes \ISR, and is the classical
reachability/shortest-path question between two prescribed independent sets in the solution space.
This variant has been studied extensively from a theoretical viewpoint, and was adopted as a metric of the CoRe Challenge~\cite{sotaokit24}.

\paragraph{Farthest variant}
Given only one independent set $I_\ini$ of size $k$, the \emph{farthest variant}~\cite{ItoKNSSTT23} aims to find an
independent set whose distance from $I_\ini$ is maximum among all
independent sets in the same connected component of $\mathcal{S}_k(G)$ containing $I_\ini$, together with that distance. Rather than connecting
two prescribed independent sets, this variant asks how far the reconfiguration
relation can carry $I_\ini$ throughout its connected component of the solution
space.

\subsection{Overview of ZDD-Based Solvers}
\label{subsec:zdd_solver}

A \emph{zero-suppressed binary decision diagram} (ZDD)~\cite{Minato1993} is a directed acyclic
graph that compactly represents a family of sets over a fixed universe
$U = \{x_1, \dots, x_n\}$ equipped with a total order $x_1 < \cdots < x_n$ on $U$.
It has two terminal nodes $\bot$ and $\top$, representing the empty family
$\emptyset$ and the family $\{\emptyset\}$, respectively, and each non-terminal
node is labeled with an element of $U$ and has a $0$-arc and a $1$-arc.
By sharing isomorphic subgraphs and suppressing redundant nodes, a ZDD can
represent a subset family of $U$ whose cardinality is exponential in $n$ using far fewer
nodes, and it supports efficient family operations such as union,
intersection, and difference through recursive procedures over its shared
structure~\cite{Minato1993,knuth:taocp41}.

For \ISR, the relevant family is the set of all independent sets of the input
graph $G = (V, E)$, where the universe is taken to be the vertex set $V$.
The state-of-the-art ZDD-based solver \textsc{ddreconf}~\cite{ItoKNSSTT23} solves
\ISR under the \TJrule rule in two phases.

In the first phase, the solver constructs the \emph{solution-space ZDD}.
It first builds a ZDD $\zind$ representing the family of all independent sets of
$G$~\cite{Hayase1995}. Recall that, under the \TJrule rule, every independent set in a
reconfiguration sequence has the same cardinality $k = |I_\ini|$ as the start
independent set $I_\ini$. The solver therefore builds a ZDD $\Z_{=k}$
representing all vertex subsets of cardinality exactly $k$~\cite{knuth:taocp41},
and obtains the solution space as $\zsol = \zind \cap \Z_{=k}$, which represents
the family of all independent sets of $G$ of cardinality $k$. The family $\sets(\zsol)$ is
exactly the node set of the solution space defined in
Section~\ref{subsec:isr}.

In the second phase, the solver repeatedly performs \emph{transition
computation} on top of $\zsol$. Starting from $\Z^{0}$ with
$\sets(\Z^{0}) = \{I_\ini\}$, it iteratively computes
\begin{equation}
  \Z^{i} \;\gets\; \mathsf{op}(\Z^{i-1}) \cap \zsol,
  \label{eq:transition}
\end{equation}
where $\mathsf{op}$ applies the \TJrule rule to every independent set encoded by
$\Z^{i-1}$. Concretely, $\mathsf{op}(\Z) = \swap(\Z, V, V)$, where
$\swap(\Z, A, R)$ represents the family
\[
  \{\, I \cup \{v\} \setminus \{v'\} \mid I \in \sets(\Z),\,
  v \in A \setminus I,\, v' \in I \cap R \,\}
\]
obtained by swapping a single token $v' \in I \cap R$ with $v \in A \setminus I$, and intersecting with $\zsol$ retains only
those sets that are independent sets of cardinality $k$. Hence $\Z^{i}$
represents exactly the family of independent sets whose distance from $I_\ini$ is
equal to $i$, and iterating \eqref{eq:transition} amounts to a breadth-first search over
the solution space. This implicit search strategy allows us to explore an
exponentially large solution space, since each $\Z^{i}$ may encode an enormous
number of independent sets within a small ZDD.

Both variants are answered from this sequence of ZDDs. For the shortest variant,
the solver checks $\Z^{i}$ after each step: once we have $I_\tar \in \sets(\Z^{i})$, the
distance between $I_\ini$ and $I_\tar$ is equal to $i$, and once we have $\Z^{i} = \bot$ without
having reached~$I_\tar$, no reconfiguration sequence exists between $I_\ini$ and $I_\tar$.
For the farthest variant, the solver continues until $\Z^{h+1} = \bot$; the last nonempty
$\Z^{h}$ then contains the independent sets farthest from $I_\ini$, and $h$ is
their distance.

The performance of this workflow is governed by the \emph{size} (number of
nodes) of the ZDDs that appear in it, which in turn depends critically on the
order in which the edges of $G$ are processed during the ZDD construction. For the
first phase, this dependence is well understood~\cite{inoue16}: the working state
maintained during top-down ZDD construction, and hence both the construction
time and the resulting ZDD size, grow rapidly with the width of the path
decomposition induced by the processing order (we make this correspondence
precise in Section~\ref{subsec:pathdecomp}). A processing order of small width
therefore keeps $\zind$, and hence $\zsol$, compact.
%; for example, the family of
%independent sets of an $8 \times 250$ grid graph has cardinality
%$3.07 \times 10^{361}$, yet it can be represented by a ZDD with only $49{,}989$
%nodes under a suitable order~\cite{ItoKNSSTT23}.
Crucially, \textsc{ddreconf}
fixes a single processing order before construction and reuses it throughout
both phases, so this order influences not only the construction of $\zsol$ but
also the entire subsequent transition computation. While the benefit of a
small-width order for the first phase is inherited from established results on
ZDD construction, whether it improves the solver as a whole, once the cost of
the transition computation and the cost of obtaining the order itself are taken
into account, is the question we study in this paper.

\section{Our Approach}
\label{sec:approach}

Our approach is built on a simple observation made in
Section~\ref{subsec:zdd_solver}: the performance of \textsc{ddreconf} is governed
by the sizes of the ZDDs that appear throughout its two phases, and these sizes
are determined by the order in which the edges of the input graph $G$ are
processed during ZDD construction. While, for the first phase, a
processing order of small width keeps the constructed ZDD small~\cite{inoue16}, it is not
immediate whether this benefit carries over to the solver as a whole, once the
cost of the subsequent transition computation and the cost of obtaining the
order itself are taken into account. The original \textsc{ddreconf} fixes the
order according to the input itself and never revisits it. We instead treat the
order as a tunable \emph{preprocessing} parameter derived from a path
decomposition of $G$, and feed the resulting order to \textsc{ddreconf} without
modifying the solver. Concretely, we compare three decompositions of increasing quality
and preprocessing cost: \OrigDecom, the decomposition induced by the input edge
order; \HeurDecom, a heuristic decomposition; and \OptDecom, a minimum-width
decomposition. This
section formalizes the path-decomposition view of the input order
(Section~\ref{subsec:pathdecomp}), describes the heuristic and exact algorithms
behind \HeurDecom and \OptDecom (Section~\ref{subsec:algorithms}), and explains
how the preprocessing is integrated into the \textsc{ddreconf} pipeline
(Section~\ref{subsec:integration}).

\subsection{Path Decomposition and the Input Edge Order}
\label{subsec:pathdecomp}

We first recall the standard definitions.
A \emph{path decomposition} of a graph $G = (V, E)$ is a sequence
$(X_1, \dots, X_\ell)$ of subsets of $V$, called \emph{bags}, such that
(i) for every edge $\{u, v\} \in E$ there is a bag $X_i$ with $u, v \in X_i$, and
(ii) for every vertex $v \in V$, the bags containing $v$ form a contiguous
subsequence. The \emph{width} of a path decomposition is
$\max_{1 \le i \le \ell} |X_i| - 1$, and the \emph{pathwidth} $\mathrm{pw}(G)$ of
$G$ is the minimum width over all its path decompositions.

The CoRe Challenge provides each instance as an input graph in the DIMACS
format~\cite{sotaokit24}: a line starting with \texttt{p} declares the numbers of vertices and
edges, and each subsequent line starting with \texttt{e} lists the two endpoints of an edge.
When \textsc{ddreconf} reads such a file and constructs $\zind$ in the first
phase, it processes the edges one by one, in the order in which they appear in
the file. The cost of this top-down construction is governed by the
\emph{frontier}, the set of vertices that are incident to both already-processed
and not-yet-processed edges; it is a folklore fact, made explicit by Inoue and
Minato~\cite{inoue16}, that the maximum frontier size over the construction
coincides with the width of a path decomposition of $G$ determined by the edge
order. Moreover, given an edge order, the width of this associated path
decomposition can be computed by the algorithm of Kobayashi and
Nakahata~\cite[Lemma~2]{KobayashiNakahata2020}. Through this correspondence, each edge order has a well-defined \emph{width},
the width of its associated path decomposition; this width is at least
$\mathrm{pw}(G)$, with equality exactly when the order is optimal.

Consequently, the order in which edges are listed in the input file, an artifact
of how each instance happens to be encoded, directly determines the width of the
path decomposition that \textsc{ddreconf} implicitly works with, and hence the
sizes of the ZDDs it builds and manipulates. For the first phase, the effect is
well understood: a smaller width keeps $\zind$, and hence $\zsol$, compact
(Section~\ref{subsec:zdd_solver}). Whether this carries over to the end-to-end
performance of the solver is the subject of this paper. This leads to a natural
plan: \emph{replace the input edge order with one derived from a path
decomposition of small width, and measure how the overall performance of
\textsc{ddreconf} changes.}

\subsection{Heuristic and Optimal Path Decomposition}
\label{subsec:algorithms}

Since computing $\mathrm{pw}(G)$ exactly is \NP-hard~\cite{ACP97}, obtaining a minimum-width path
decomposition is expensive in general. Besides the decomposition induced by the
input order (\OrigDecom), we obtain a path decomposition using the algorithm of
Kobayashi, Komuro, and Tamaki~\cite{KobayashiKomuroTamaki2014}. This algorithm
computes an optimal path decomposition by reducing the search space through
commitments and gradually narrowing down to minimum width; it produces a first
feasible decomposition quickly and, if run to completion, eventually returns an
optimal one. We exploit both ends of this process, yielding two path
decompositions of different quality and cost. This algorithm is an existing
method, which we use as is and do not claim as a contribution of this paper.

\paragraph{Heuristic decomposition (\HeurDecom)}
As a fast preprocessing step, we take the first decomposition that the algorithm
produces, whose width is small but not necessarily minimum. Obtaining it is
quick enough to serve as a lightweight preprocessor even on the larger CoRe
Challenge instances. Because this first decomposition is not guaranteed to attain
the minimum width, it is occasionally wider than the one induced by the original
input order. Since the width of \OrigDecom is itself available
(Section~\ref{subsec:pathdecomp}), we compare the two and keep the better one:
only when the first decomposition is strictly wider than the original do we fall
back to the original order. Thus \HeurDecom never yields a wider decomposition
than \OrigDecom.

\paragraph{Optimal decomposition (\OptDecom)}
To examine the effect of attaining the true minimum width, we run the same
algorithm to completion, obtaining an optimal path decomposition, i.e., one of
width exactly $\mathrm{pw}(G)$. This is far more time-consuming than obtaining the
first decomposition, and is prohibitively so for some instances, but it lets us
measure the best possible gain achievable through path-decomposition-based
reordering and compare it against \HeurDecom.

\subsection{Integration with \textsc{ddreconf} and the Three Decompositions}
\label{subsec:integration}

In all three cases, the path decomposition is used purely as preprocessing: from
the decomposition we derive an edge order, write it back as a reordered DIMACS
file, and run the unmodified \textsc{ddreconf} on it. The solver itself is
identical across all runs, including its solution-space ZDD construction, its
swap-based transition computation, and its breadth-first reconfiguration search;
only the order in which the edges of $G$ are presented differs. This makes the
comparison clean: any difference in running time, memory usage, or
solved-instance count is attributable solely to the edge order, and hence to the
underlying path decomposition.

We compare the following three path decompositions.
\begin{description}
  \item[\OrigDecom:] the path decomposition induced by the edge order as given in
    the CoRe Challenge input file, used without any reordering. This is the
    baseline and corresponds to the original \textsc{ddreconf} behavior.
  \item[\HeurDecom:] a path decomposition of small but not necessarily minimum
    width, computed by the heuristic algorithm~\cite{KobayashiKomuroTamaki2014};
    if it is strictly wider than \OrigDecom, we fall back to the original order,
    so that \HeurDecom is never wider than \OrigDecom.
  \item[\OptDecom:] an optimal (minimum-width) path decomposition, computed by an
    exact algorithm.
\end{description}
For \HeurDecom and \OptDecom, we derive an edge order from the computed path
decomposition and run \textsc{ddreconf} on that order. The preprocessing cost
increases sharply along this list: \OrigDecom incurs none, \HeurDecom adds only
a fast preprocessing step, whereas \OptDecom can be extremely time-consuming.
Section~\ref{sec:experiments} reports how these three path decompositions compare
in practice, and in particular whether the inexpensive \HeurDecom decomposition
already captures most of the benefit that the costly \OptDecom decomposition
provides.


\section{Experimental Results}
\label{sec:experiments}

\subsection{Experimental Setup}
\label{subsec:setup}

On top of \textsc{ddreconf}~\cite{ItoKNSSTT23} we implemented the
path-decomposition preprocessing and compared the three decompositions
\OrigDecom, \HeurDecom, and \OptDecom of Section~\ref{sec:approach}.

All experiments were run on a Linux OS with a 2.3 GHz Intel Xeon Gold processor and 2TB memory.

\paragraph{Benchmarks}
Our instances come from the CoRe Challenge 2022 and 2023~\cite{sotaokit24}:
$693$ instances in all, drawn from $519$ distinct input graphs, where a single
graph may give rise to several instances that differ only in their start and goal
independent sets. The graphs vary widely in size, as shown in
\figurename~\ref{fig:graphs226}, which plots all $519$ graphs by their numbers of
vertices and edges.

\paragraph{Time limits}
We budget preprocessing and solving separately. \OrigDecom uses the input order
as given and requires no preprocessing, whereas \HeurDecom and \OptDecom are each
computed under a one-hour preprocessing budget. \OptDecom whose preprocessing does
not finish within the hour is left undefined for that instance. 
If the heuristic does not finish in time, we set the width of \HeurDecom to $+\infty$; 
the fallback rule then applies to this case, and \HeurDecom coincides with \OrigDecom.
We then pass the (possibly reordered) input to the unmodified
\textsc{ddreconf} and allow it $30$ minutes of wall-clock time per instance; an
instance counts as \emph{solved} under a decomposition if \textsc{ddreconf}
returns the correct answer within that limit.

\section{Analyses}
\label{sec:analyses}

The previous section left us with two observations: reordering the input edges
along a path decomposition lets \textsc{ddreconf} solve more instances, and the width of that decomposition affects solvability. We now ask where these gains come from.
The place to look is the ZDDs the solver builds, so we follow the effect of width
through both phases of \textsc{ddreconf}: the initial construction of the
solution-space ZDD $\zsol$, and the transition computations that rebuild a ZDD $\Z^i$ at every step $i$ of the search.

\subsection{The First Phase: Building the Solution-Space ZDD}
\label{subsec:zddsize}

We start with the first phase, the solution-space ZDD $\zsol$. To put the three
decompositions on equal footing we restrict attention to the instances that all
three solve within the time limit ($210$ on the shortest variant, $208$ on the
farthest), so that each decomposition yields a value for every instance.
Table~\ref{tab:zddsize} reports, for each decomposition and variant, the geometric
mean and the maximum of the width, the solution-space ZDD size (the number of ZDD nodes), the first-phase
construction time, the total solving time, and the peak memory over these
instances.

The solution-space ZDD shrinks steadily as the width falls, and the gap widens
sharply on the hardest instances. Its geometric-mean size drops from $3{,}011$
nodes under \OrigDecom to $1{,}191$ under \HeurDecom and $971$ under \OptDecom.
Construction time and peak memory follow the same pattern, as
Table~\ref{tab:zddsize} shows. This is the behavior predicted by the analysis of
the first phase~\cite{inoue16}: across these instances the Spearman correlation
between width and ZDD size is $0.66$, confirming experimentally the link between
width and ZDD size that motivates the reordering. 
As with the widths themselves,
\HeurDecom recovers most of what \OptDecom offers, reaching about $80\%$ of the
optimal geometric-mean reduction in ZDD size.

This aggregate picture has exceptions at the level of individual instances: a
narrower decomposition does not always yield a smaller ZDD, because the width of a
decomposition only gives an upper bound on the ZDD size~\cite{inoue16}, which still
leaves the construction free to produce ZDDs of different sizes. On about $37\%$ of
the commonly-solved instances, the ZDD under \OptDecom is in fact slightly larger
than under \OrigDecom. But the difference is marginal, at most $7\%$ in size and at
most $120$ nodes in absolute terms.
Every such case has a small ZDD, of at most
$2{,}136$ nodes under \OrigDecom. 
We note that, across the commonly-solved instances, the ZDD under
\OrigDecom ranges from $8$ to $1.4\times10^{8}$ nodes, and hence every reversal sits at the
very bottom of this range. 
Therefore, where the construction cost actually matters, no such
reversal occurs and \OptDecom always yields the smallest ZDD. 
Therefore, narrower decompositions never lose ground where it counts, which is why the
geometric mean and the maximum in Table~\ref{tab:zddsize} favor them so clearly.

\subsection{The Second Phase: Rebuilding the ZDD at Each Step}
\label{subsec:secondphase}
What is specific to reconfiguration is the second phase, where the solver rebuilds a
ZDD $\Z^i$ at every step of the search; this is where we now look. Each $\Z^i$
represents a slice of the solution space $\zsol$, and hence a natural expectation is
that a narrower decomposition shrinks the intermediate ZDDs as well. The question is
whether this expectation holds, and how the effect compares with the one in the
first phase. We measure the second phase by the peak intermediate ZDD size,
$\max_i|\Z^i|$, and we restrict attention to the commonly-solved instances ($210$ on
the shortest variant, $208$ on the farthest), where every run reaches the end of its
search and the peak is the true maximum rather than a value cut off by the time
limit.

The expectation broadly holds. The first-phase and second-phase reductions move
together: across the commonly-solved instances, the Spearman correlation between the
$\zsol$ reduction and the peak-$|\Z^i|$ reduction (\OrigDecom over \OptDecom) is
$0.64$. Nonetheless, the effect is weaker in the second phase than in the first. Over
the commonly-solved instances, the $\zsol$ reduction has a geometric mean of
$3.1\times$, but the peak-$|\Z^i|$ reduction only $1.4\times$, and on $80\%$ of the
instances the second-phase reduction is no larger than the first.
Figure~\ref{fig:sp-scatter} shows this directly: the points lie below the diagonal.
In particular, every instance whose first-phase reduction is below $1.2\times$ also
has a second-phase reduction below $1.4\times$. 
This implies that when the \OrigDecom decomposition builds a small $\zsol$, it can also build small intermediate ZDDs.

However, a narrower decomposition still brings a clear benefit; specifically, where the first phase gains a lot, so does the second. 
On the instances where $\zsol$ shrinks by at least $10\times$, the peak intermediate ZDD shrinks by a
median of about $5\times$, and a narrower decomposition keeps its advantage through
the search. Figure~\ref{fig:sp-traj} illustrates this on \textsf{mug88\_1}: the peak
intermediate ZDDs under \OrigDecom, \HeurDecom, and \OptDecom are $1.4\times10^{6}$,
$3.7\times10^{4}$, and $5.0\times10^{3}$ nodes, respectively.

\subsection{Reconfiguration Length}
\label{subsec:reconflength}

One distinctive strength of the ZDD-based approach is its reach into very long
reconfiguration sequences. By \PSPACE-completeness, some instances have a shortest
reconfiguration sequence of super-polynomial length unless $\PSPACE = \NP$
(Section~\ref{subsec:isr}). Indeed, even if we restrict to only the instances solved
within the time limit, the shortest reconfiguration lengths in the benchmark span
from $1$ to about $2.2\times10^{5}$. Figure~\ref{fig:reconf-length} plots, for every
instance solved under \HeurDecom, the end-to-end solving time against the shortest
reconfiguration length $\ell$, coloring each point by the width of its \HeurDecom
decomposition.

Two patterns are visible in \figurename~\ref{fig:reconf-length}. First, the long
sequences, up to $\ell\approx2.2\times10^{5}$, are solved only at small width: every
solved instance with $\ell\geq100$ has width at most $23$, with a median of $10$,
whereas no instance of large width reaches such lengths within the limit. These
long-sequence instances are solved cheaply, and their solving time does not blow up
exponentially with $\ell$: on the log--log axes, their points do not curve upward
but follow a nearly straight trend, so the length enters the cost only polynomially.
Second, the short sequences are solved across a wide range of widths, including
large ones. Here the solving time grows with the width: the widest instances are the
slowest, taking close to the $30$-minute limit.

\section{Conclusion}
\label{sec:conclusion}

We studied how reordering the input edges along a path decomposition of small width affects the ZDD-based solver \textsc{ddreconf}: with no change to the solver, it solves more instances. 
The benefit of a smaller width is known for the first phase, constructing the solution space $\zsol$. 
We showed that it appears in the second phase as well, which is specific to reconfiguration, where the solver rebuilds a ZDD at every reconfiguration step. 
We emphasize that the heuristic decomposition recovers most of the benefit of the optimal one at a small fraction of the cost, making it a compelling choice in practice.
