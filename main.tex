\begin{abstract}
Zero-suppressed binary decision diagrams (ZDDs) compactly represent families of
combinatorial solutions, and it is well known that building a ZDD along a path
decomposition of small width keeps it small. Solvers for combinatorial
reconfiguration, however, use a ZDD in a more demanding way: they first build a
ZDD of the entire solution space and then repeatedly rebuild ZDDs as the search
advances, one transition step at a time. Whether, and how much, a small-width
decomposition helps such a solver overall has been unclear, once the cost of
these repeated computations and of computing the decomposition itself is taken
into account.
We study path decomposition as a preprocessing step for \textsc{ddreconf}, a
state-of-the-art ZDD-based solver for independent set reconfiguration under the
token jumping rule. The preprocessing only reorders the input edges along a
small-width decomposition; the solver itself is left untouched. On both the
shortest and the farthest variant of the problem, the reordering consistently
reduces ZDD size, memory, and running time, and lets the solver clear more
instances within the time limit. Computing an optimal decomposition turns out to
be unnecessary: a fast greedy reaches essentially the same width and the same
gains, and on large instances, whose optimal pathwidth is out of reach, it is the
only option that lets the solver finish at all. Tracing the effect of width
through the solver, we find that it shapes not only the solution-space ZDD but
also the intermediate ZDDs rebuilt at every step, so its benefit extends across
the whole search; and the width of the decomposition, rather than the size of the
graph, turns out to be a strong predictor of whether an instance can be solved.
\end{abstract}

\section{Introduction}
Combinatorial reconfiguration asks how one feasible solution of a combinatorial
problem can be turned into another through a sequence of small local changes, with
every intermediate step also feasible~\cite{nishimura18}. A prototypical example
is independent set reconfiguration (\ISR): given two independent sets of a graph,
can one be transformed into the other by moving the tokens placed on its vertices,
keeping an independent set throughout? We adopt the token jumping rule, in which
each step moves a single token to another vertex, equivalently, swaps one vertex
of the current independent set for a vertex outside it. \ISR is \PSPACE-complete in
general~\cite{kamemi12,itdehapasiueun11}, and building solvers that are efficient
in practice remains an active line of work.

Figure~\ref{fig:core-example} illustrates a benchmark instance from the CoRe
Challenge 2023~\cite{sotaokit24}: each node of the solution-space graph on the
right is an independent set of size $3$, and two nodes are adjacent when one can
be reached from the other by a single token jump.

To address this computational hardness, various practical approaches have been
developed, including Answer Set Programming (ASP)-based
solvers~\cite{ybisu24} and AI-planning techniques~\cite{cherkamupeposesisp23},
both of which excel at finding short reconfiguration sequences. Another line of
work uses zero-suppressed binary decision diagrams (ZDDs)~\cite{Minato1993}, compact data structures
for implicitly representing families of sets. By using ZDDs to compress
structural regularity in the solution space, ZDD-based solvers provide a
complementary approach and can sometimes outperform ASP and planning methods,
especially on instances whose solution spaces admit compact ZDD representations~\cite{ItoKNSSTT23}.
When a ZDD is built from a graph, its size is
closely tied to the pathwidth of the graph: roughly speaking, the smaller the
pathwidth, the smaller the ZDD~\cite{inoue16}. This connection is the basis of a
standard speedup in many ZDD applications, namely ordering variables along a
small-width path decomposition to keep the constructed ZDD small.


%Reconfiguration solvers, however, use a ZDD in a two-stage way that complicates
In such solvers, this representation is used in a two-stage way, which complicates
the picture. They first construct a ZDD representing the entire solution space,
and then repeatedly apply transition operations to the diagram, advancing the search one
step at a time until a reconfiguration sequence is found. A small-width
decomposition is known to help the first stage, the construction of the
solution-space ZDD. It has remained unclear how much it helps the solver as a
whole, once one also accounts for the cost of the repeated transition computations
and for the cost of computing the decomposition in the first place.



We study path decomposition as a preprocessing step for
\textsc{ddreconf}~\cite{ItoKNSSTT23}, an implementation of a state-of-the-art ZDD-based
solver for \ISR
under the token jumping rule. \textsc{ddreconf} reads a graph as an ordered list
of edges and processes them in that order; this order fixes the path
decomposition the solver implicitly follows, and hence the sizes of the ZDDs it
builds. Our preprocessing simply reorders the edges along a path decomposition of
small width and leaves the solver itself untouched. We evaluate the idea on two
variants that probe the solution space in complementary ways: the shortest
variant, which looks for a shortest reconfiguration sequence between two given
independent sets, and the farthest variant, which looks for an independent set as
far as possible from a given one. Our contributions are as follows.
\begin{itemize}
\item We show experimentally that reordering the input edges along a path
decomposition consistently improves \textsc{ddreconf}: it shrinks the
solution-space ZDD, lowers running time and memory, and increases the number of
instances solved within the time limit, on both the shortest and the farthest
variant.
\item We show that an optimal decomposition is unnecessary in practice. A fast
greedy decomposition reaches essentially the same width as the optimal one and
brings the same gains; moreover, on large instances whose optimal pathwidth cannot
be computed in reasonable time, the greedy is the only one of the two that lets
the solver reach them at all.
\item We trace where the gains come from and find that the decomposition width
shapes not only the solution-space ZDD but also the intermediate ZDDs that are
rebuilt at every transition step, so its effect extends across the whole search.
The exceptions form a recognizable class of structured instances whose pathwidth
is a loose upper bound on the actual ZDD size.
\item We find that the width of the decomposition the solver works with, rather
than the size of the input graph, is a strong predictor of whether an instance can
be solved.
\end{itemize}

\subsection{Independent Set Reconfiguration}
\label{subsec:isr}

Let $G=(V,E)$ be an unweighted and undirected graph, where $V$ is the set of vertices and $E$ is the set of edges. 
A vertex subset $I\subseteq V$ is an \emph{independent set of $G$} if no two vertices in $I$ are adjacent in $G$.
For a positive integer $k$, the \emph{solution-space graph} $\solgraph{k}{G}$ is a graph whose nodes are the independent sets of $G$ with size exactly $k$; two nodes are joined by an edge in $\solgraph{k}{G}$ if and only if the corresponding independent sets $I$ and $I'$ satisfy $|I \setminus I'| = |I' \setminus I| = 1$. (See \figurename~\ref{fig:core-example}.) 
Then, a path in $\solgraph{k}{G}$ connecting two nodes is called a \emph{reconfiguration sequence} between them, and the \emph{length} of the reconfiguration sequence is defined to be the number of edges in the path. 
The \emph{distance} between two nodes in $\solgraph{k}{G}$ is the length of a shortest reconfiguration sequence between them;
or it is $+\infty$ if no such sequence exists. 

Given a graph $G$ and two independent sets $I_{\ini}$ and $I_{\tar}$ of $G$ such that $|I_{\ini}|=|I_{\tar}| = k$, the \emph{Independent Set Reconfiguration} (\ISR) problem under the \emph{Token Jumping} (TJ) rule is to determine whether there exists a reconfiguration sequence between $I_{\ini}$ and $I_{\tar}$ in $\solgraph{k}{G}$.
Note that \ISR is a decision problem, and hence we are not asked for an actual reconfiguration sequence as an output. 
This problem is known to be \PSPACE-complete~\cite{kamemi12}, even when restricted to graphs of bounded pathwidth~\cite{Wrochna18}.
The \PSPACE-completeness implies that there are yes-instances such that even a
shortest reconfiguration sequence has super-polynomial length, unless $\PSPACE = \NP$.

In this paper, we study two variants of \ISR under \TJrule which probe the structure of the solution-space graph in complementary ways.

\paragraph{Shortest variant} 
Given an \ISR instance $(G, I_\ini, I_\tar)$, the \emph{shortest variant} aims to
determine the existence of a reconfiguration sequence between $I_\ini$ and
$I_\tar$ and, if so, to compute the distance between them. 
This variant generalizes \ISR, and is the classical
reachability/shortest-path question between two prescribed independent sets in the solution-space graph.
This variant has been studied extensively from a theoretical viewpoint, and was adopted as a metric of the CoRe Challenge~\cite{sotaokit24}.

\paragraph{Farthest variant}
Given only one independent set $I_\ini$ of size $k$, the \emph{farthest variant}~\cite{ItoKNSSTT23} aims to find an
independent set whose distance from $I_\ini$ is maximum among all
independent sets in the same connected component of $\solgraph{k}{G}$ containing $I_\ini$, together with that distance. Rather than connecting
two prescribed independent sets, this variant asks how far the reconfiguration
relation can carry $I_\ini$ throughout its connected component of the solution
space.

\subsection{Zero-suppressed binary decision diagrams}
\label{subsec:zdd_def}

A \emph{zero-suppressed binary decision diagram} (ZDD)~\cite{Minato1993,knuth:taocp41} is a directed acyclic
graph that compactly represents a family of sets over a fixed universe
$U = \{x_1, x_2, \dots, x_n\}$ equipped with a total order $x_1 < x_2 < \cdots < x_n$ on $U$.
The ZDD $\Z$ has two terminal nodes $\bot$ and $\top$, and each non-terminal
node is labeled with an element of $U$ and has a $0$-arc and a $1$-arc, both pointing to nodes with larger labels. 
$\Z$ also has a single root, namely, a node whose in-degree is zero.
The family $\sets(\Z)$ represented by $\Z$ is read off from the paths on $\Z$: every path
from the root to $\top$ encodes the set consisting of the labels of the nodes
that the path leaves through their $1$-arcs, and $\sets(\Z)$ is the collection of
the sets encoded by all such paths. 

For example, to represent a family of independent sets in the input graph $G = (V,E)$, the universe is taken to be the vertex set $V$.
Figure~\ref{fig:zdd-order} shows two ZDDs that represent the same family, the independent sets of size $2$ in the path $P_5 = v_1 v_2 v_3 v_4 v_5$; in other words, each ZDD represents the node set of the solution-space graph $\solgraph{2}{P_5}$.

By considering equivalent substructures and suppressing redundant nodes, a ZDD can
represent a subset family of $U$ whose cardinality can be exponential in $|U|$
using far fewer nodes, and it supports efficient family operations such as union,
intersection, and difference~\cite{Minato1993,knuth:taocp41}. The running time of these operations depends on the sizes of the ZDDs involved,
where the \emph{size} of a ZDD is the number of its nodes. The size is in turn affected
by the total order on $U$: two ZDDs representing the same family
may differ in size under different orders, as Figure~\ref{fig:zdd-order} shows.

\subsection{Overview of ZDD-Based Solvers}
\label{subsec:zdd_solver}

The state-of-the-art ZDD-based solver \textsc{ddreconf}~\cite{ItoKNSSTT23} solves the shortest and farthest variants of \ISR in two phases.
Let $G = (V,E)$ be an input graph, and let $k = |I_\ini|$ for a given start independent set $I_\ini$ of $G$. 

In the first phase, the solver constructs the \emph{solution-space ZDD} $\zsol$ representing the node set of the solution-space graph $\solgraph{k}{G}$.
To do so, the solver first builds a ZDD $\zind$ representing the family of all independent sets of
$G$~\cite{Hayase1995}, and a ZDD $\Z_{=k}$ representing the family of all vertex subsets of cardinality exactly $k$~\cite{knuth:taocp41}. 
Then, the solver obtains $\zsol$ by the intersection operation $\zsol = \zind \cap \Z_{=k}$.

In the second phase, the solver repeatedly performs \emph{transition computation}
on top of $\zsol$, and obtains a ZDD $\Z^{i}$ representing the family of all
independent sets of $G$ whose distance from $I_\ini$ is equal to $i$.
For a ZDD $\Z$ representing a family of independent sets of $G$, we define $\mathsf{TJ}(\Z)$ as a ZDD representing the family of vertex subsets of $G$ (which are not necessarily independent sets) obtained by applying the \TJrule rule to every independent set encoded
by $\Z$, that is, it represents the family
$\{\, I \cup \{v\} \setminus \{v'\} \mid I \in \sets(\Z),\,
  v \in V \setminus I,\, v' \in I \,\}$. 
Then, starting from $\Z^{0}$ with $\sets(\Z^{0}) = \{I_\ini\}$, the solver iteratively computes 
$\Z^{i} \;\gets\; \bigl(\mathsf{TJ}(\Z^{i-1}) \cap \zsol\bigr) \setminus \bigcup_{j=0}^{i-1} \Z^{j}$.

Both variants are answered from this sequence of ZDDs. For the shortest variant,
the solver checks $\Z^{i}$ after each step.
Once $I_\tar \in \sets(\Z^{i})$ holds, the distance between $I_\ini$ and $I_\tar$ is equal to $i$; 
and once $\Z^{i}$ becomes empty, that is, $\sets(\Z^{i}) = \emptyset$, before $I_\tar$ is reached, no
reconfiguration sequence exists between $I_\ini$ and $I_\tar$.
For the farthest variant, the solver continues until $\Z^{\ell+1}$ becomes empty;
the last nonempty $\Z^{\ell}$ then contains the independent sets farthest from
$I_\ini$, and $\ell$ is their distance.

The performance of this workflow is governed by the sizes of the ZDDs that appear
in it. Crucially, \textsc{ddreconf} fixes a single total order on the vertex set $V$ before
construction and reuses it throughout both phases.
Therefore, the order influences not
only the construction of $\zsol$ but also the entire subsequent transition
computation.

\section{Our Approach}
\label{sec:approach}
As described in Section~\ref{subsec:zdd_solver}, \textsc{ddreconf} fixes a single
total order on the vertex set $V$ and uses it throughout its two phases. Our approach is to
control this order through a path decomposition of the input graph $G$. 
As we will formalize in Section~\ref{subsec:pathdecomp}, a path decomposition of $G$
gives a vertex order, and it is known that the width of the decomposition gives an
upper bound on the size of the solution-space ZDD $\zsol$ constructed under this
order~\cite{inoue16}. This suggests that an order obtained from a smaller-width decomposition would yield a smaller $\zsol$ in the first phase.
However, what happens in the
second phase is far less clear. There, a ZDD is rebuilt at every step of the
search, and it is not obvious that a small width still helps. This is the
question we pursue: if we feed \textsc{ddreconf} a vertex order obtained from a
path decomposition of small width, do the ZDDs stay small in the second phase
as well, and does this make the solver as a whole more efficient? 
To investigate this question, we treat the vertex order as a tunable
\emph{preprocessing} parameter derived from a path decomposition of $G$, and feed
the resulting order to \textsc{ddreconf} without modifying the solver itself.

\subsection{Path Decomposition and Vertex Order}
\label{subsec:pathdecomp}
We first recall the standard definitions.
A \emph{path decomposition} of a graph $G = (V, E)$ is a sequence
$(X_1, X_2, \dots, X_q)$ of subsets of $V$, called \emph{bags}, such that
\begin{enumerate}
\item[(i)] for every edge $uv \in E$, there is a bag $X_i$ such that $u, v \in X_i$; and
\item[(ii)] for every vertex $v \in V$, the bags containing $v$ form a contiguous
subsequence.
\end{enumerate}
The \emph{width} of a path decomposition is
$\max_{1 \le i \le q} |X_i| - 1$, and the \emph{pathwidth} $\mathsf{pw}(G)$ of
$G$ is the minimum width over all its path decompositions.
For a given graph $G$, the problem of computing $\mathsf{pw}(G)$ is
\NP-hard~\cite{ACP97}.

The CoRe Challenge~\cite{sotaokit24} provides an input graph $G=(V,E)$ in the DIMACS
format: a line starting with \texttt{p} declares the numbers of vertices and
edges, and each subsequent line starting with \texttt{e} lists the two endpoints
of an edge. Therefore, a DIMACS file fixes an order on the edges of $G$.
Such an edge order naturally induces a path decomposition of
$G$~\cite[Lemma~2]{KobayashiNakahata2020}, whose width is then read off from the
sizes of its bags. 

A path decomposition in turn gives a vertex order: order the vertices by the
first bag in which they appear, breaking ties arbitrarily. 
When \textsc{ddreconf} reads a DIMACS file, it processes the edges in the order
in which they appear.
As a result, \textsc{ddreconf} uses the total order on $V$, which coincides with
the vertex order given by the path decomposition induced by the DIMACS file.

In this way, the input DIMACS file fixes the path decomposition (and its width)
that \textsc{ddreconf} implicitly works with.

decomposition of small width, and measure how the overall performance of

\subsection{Three Path Decompositions}
\label{subsec:algorithms}

Recall that the width of a path decomposition gives an upper bound on the size of the solution-space ZDD $\zsol$ constructed in the first phase~\cite{inoue16}.
Therefore, we would
like to feed \textsc{ddreconf} a DIMACS file whose induced path decomposition
has small width. To obtain such a file, we first compute a path decomposition of
$G$ of small width and then encode it as a DIMACS file.
Thus, the question is how to compute a path decomposition of small width.

Since the problem of computing $\mathsf{pw}(G)$ is \NP-hard~\cite{ACP97}, obtaining a
minimum-width path decomposition is expensive in general.
We nevertheless employ the algorithm of Kobayashi, Komuro, and Tamaki~\cite{KobayashiKomuroTamaki2014},
which produces a first feasible decomposition quickly and gradually narrows down
to the minimum width; if run to completion, it returns an optimal
decomposition. We exploit both ends of this process, yielding two path
decompositions of different quality and cost. 

Together with the path decomposition induced by the input DIMACS file itself, we compare the
following three path decompositions in this paper.

\paragraph{Original decomposition (\OrigDecom)}
We take the path decomposition induced by the CoRe Challenge input
file as given, without any preprocessing. This is the baseline and
corresponds to the original behavior of \textsc{ddreconf}.

\paragraph{Greedy decomposition (\HeurDecom)}
We take the first path decomposition produced by the
algorithm~\cite{KobayashiKomuroTamaki2014}.
The decomposition is constructed in a greedy way, and is not guaranteed to attain
the minimum width.
Indeed, it is occasionally wider than \OrigDecom.
We compare the two widths and keep the better one:
we fall back to the original file only when the first decomposition is strictly
wider than \OrigDecom. 
Thus, \HeurDecom is never wider than \OrigDecom.

\paragraph{Optimal decomposition (\OptDecom)}
We take an optimal path decomposition, of width exactly $\mathsf{pw}(G)$, which
the algorithm~\cite{KobayashiKomuroTamaki2014} returns if it is run to
completion. 


\subsection{Using the Three Decompositions with \textsc{ddreconf}}
\label{subsec:integration}

For \HeurDecom and \OptDecom, the computed path decomposition is used purely as
preprocessing: we encode it as a reordered DIMACS file and run the unmodified
\textsc{ddreconf} on that file. For \OrigDecom, the solver simply runs on the
input file as given. The solver itself is identical across all runs; only the input file differs. 
Therefore, any difference in running time, memory usage, or the number of solved instances is attributable to the input file alone.

The preprocessing cost increases sharply along the three decompositions.
\OrigDecom incurs none. \HeurDecom adds a step that typically takes only a few
seconds. \OptDecom, in contrast, is far more time-consuming, and prohibitively
so for some instances; it nevertheless serves as a reference point, telling us
the gain attainable when the width is the minimum possible, against which the
inexpensive \HeurDecom can be compared.
Section~\ref{sec:experiments} reports how the three decompositions compare in
practice, and in particular whether the inexpensive \HeurDecom already captures
most of the benefit that the costly \OptDecom provides.

\section{Analyses}
\label{sec:analyses}

The previous section showed that a narrower decomposition lets \textsc{ddreconf} solve more instances. 
We now ask where these gains come from.
The place to look is the ZDDs the solver builds, so we follow the effect of width
through both phases of \textsc{ddreconf}: the initial construction of the
solution-space ZDD $\zsol$, and the transition computations that rebuild a ZDD $\Z^i$ at every step $i$ of the search.

\subsection{The First Phase: Building the Solution-Space ZDD}
\label{subsec:zddsize}

We start with the first phase, the solution-space ZDD $\zsol$. To put the three
decompositions on equal footing, we restrict attention to the instances that all
three solve within the time limit ($210$ on the shortest variant, $208$ on the
farthest), so that each decomposition yields a value for every instance.
Table~\ref{tab:zddsize} reports, for each decomposition and variant, the geometric
mean and the maximum of the width, the solution-space ZDD size (the number of ZDD nodes), the first-phase
construction time, the total solving time, and the peak memory over these
instances.

The solution-space ZDD shrinks steadily as the width falls, and the gap widens
sharply on the hardest instances. Its geometric-mean size drops from $3{,}011$
nodes under \OrigDecom to $1{,}191$ under \HeurDecom and $971$ under \OptDecom.
Construction time and peak memory follow the same pattern, as
Table~\ref{tab:zddsize} shows. This is the behavior predicted by the analysis of
the first phase~\cite{inoue16}: across these instances the Spearman correlation
between width and ZDD size is $0.66$, confirming experimentally the link between
width and ZDD size that motivates choosing a narrower decomposition. 
As with the widths themselves,
\HeurDecom recovers most of what \OptDecom offers, reaching about $80\%$ of the
optimal geometric-mean reduction in ZDD size.

This aggregate picture has exceptions at the level of individual instances: a
narrower decomposition does not always yield a smaller ZDD, because the width of a
decomposition only gives an upper bound on the ZDD size~\cite{inoue16}, which still
leaves the construction free to produce ZDDs of different sizes. On about $37\%$ of
the commonly-solved instances, the ZDD under \OptDecom is in fact slightly larger
than under \OrigDecom. But the difference is marginal, at most $7\%$ in size and at
most $120$ nodes in absolute terms.
Every such case has a small ZDD, of at most
$2{,}136$ nodes under \OrigDecom. 
We note that, across the commonly-solved instances, the ZDD under
\OrigDecom ranges from $8$ to $1.4\times10^{8}$ nodes, and hence every reversal sits at the
very bottom of this range. 
Therefore, where the construction cost actually matters, no such
reversal occurs and \OptDecom always yields the smallest ZDD. 
This is why the geometric mean and the maximum in Table~\ref{tab:zddsize} favor them so clearly.

\subsection{The Second Phase: Rebuilding the ZDD at Each Step}
\label{subsec:secondphase}
What is specific to reconfiguration is the second phase, where the solver rebuilds a
ZDD $\Z^i$ at every step $i$ of the search; this is where we now look. Each $\Z^i$
represents a slice of the solution space $\zsol$, and hence a natural expectation is
that a narrower decomposition shrinks the intermediate ZDDs as well. 
We measure the second phase by the peak intermediate ZDD size,
$\max_i|\Z^i|$, and we restrict attention to the commonly-solved instances ($210$ on
the shortest variant, $208$ on the farthest), where every run reaches the end of its
search and the peak is the true maximum rather than a value cut off by the time
limit.

The expectation broadly holds. For each instance, we define the \emph{shrink
factor} of a ZDD as its size under \OrigDecom divided by its size under
\OptDecom; a factor above $1$ means that \OptDecom yields a smaller ZDD than
\OrigDecom does.
The first-phase and second-phase factors move together: across the commonly-solved instances, the
Spearman correlation between the shrink factor of $\zsol$ and that of the peak
$|\Z^i|$ is $0.64$. Nonetheless, the effect is weaker in the second phase than in
the first. Over the commonly-solved instances, $\zsol$ shrinks by a factor of
$3.1$ in geometric mean, whereas the peak $|\Z^i|$ shrinks only by a factor of
$1.4$, and on $80\%$ of the instances the second-phase factor is no larger than
the first. Figure~\ref{fig:sp-scatter} shows this directly: the points lie below
the diagonal. 

However, a narrower decomposition still brings a clear benefit; specifically,
where the first phase gains a lot, so does the second. On the instances whose
$\zsol$ has a shrink factor of at least $10$, the peak intermediate ZDD has a
median shrink factor of about $5$, and a narrower decomposition keeps its
advantage through the search. Figure~\ref{fig:sp-traj} illustrates this on
\textsf{mug88\_1}: the peak intermediate ZDDs under \OrigDecom, \HeurDecom, and
\OptDecom are $1.4\times10^{6}$, $3.7\times10^{4}$, and $5.0\times10^{3}$ nodes,
respectively.

\subsection{Reconfiguration Length}
\label{subsec:reconflength}

One distinctive strength of the ZDD-based approach is its reach into very long
reconfiguration sequences. By \PSPACE-completeness, some instances have a shortest
reconfiguration sequence of super-polynomial length unless $\PSPACE = \NP$
(Section~\ref{subsec:isr}). Indeed, even if we restrict to only the instances solved
within the time limit, the shortest reconfiguration lengths in the benchmark span
from $1$ to about $2.2\times10^{5}$. Figure~\ref{fig:reconf-length} plots, for every
instance solved under \HeurDecom, the end-to-end solving time against the shortest
reconfiguration length $\ell$, coloring each point by the width of its \HeurDecom
decomposition.

Two patterns are visible in \figurename~\ref{fig:reconf-length}. First, the long
sequences, up to $\ell\approx2.2\times10^{5}$, are solved only at small width: every
solved instance with $\ell\geq100$ has width at most $23$, with a median of $10$,
whereas no instance of large width reaches such lengths within the limit. 
These long-sequence instances are solved cheaply, and their solving time does not blow up
exponentially with $\ell$. 
%: on the log--log axes, their points do not curve upward
%but follow a nearly straight trend, so the length enters the cost only polynomially.
Second, the short sequences are solved across a wide range of widths, including
large ones. Here the solving time grows with the width: the widest instances are the
slowest, taking close to the $30$-minute limit.

\section{Conclusion}
\label{sec:conclusion}

We studied how choosing a path decomposition of small width affects the ZDD-based solver \textsc{ddreconf}: with no change to the solver itself, it solves more instances. 
The benefit of a smaller width is known for the first phase, constructing the solution space $\zsol$. 
We showed that it appears in the second phase as well, which is specific to reconfiguration, where the solver rebuilds a ZDD at every reconfiguration step. 
We emphasize that the greedy decomposition recovers most of the benefit of the optimal one at a small fraction of the cost, making it a compelling choice in practice.
